Project State & Implementation Roadmap

From dashboard demo to enterprise operating layer.

This page documents the current state, future state, implementation path, assumptions, dependencies, risks, feasibility, and projected rollout timeline for turning the current dashboard demo into a practical V1 operating dashboard. The first implementation path is intentionally lightweight: host the dashboard from Commit's GitHub, embed it in the SharePoint IT Team Hub, use Google Sheets as the data source, and use AppleScript to refresh the dashboard from intake and update form submissions.

Current State V1 Build Candidate

Static HTML demos show the user experience. The V1 path is to convert the demo into a GitHub-hosted dashboard embedded in SharePoint and refreshed from Google Sheets.

Feasibility High for V1

Practical as a lightweight internal dashboard using GitHub Pages, SharePoint embed, Google Sheets, intake/update forms, and AppleScript automation.

Recommended Lift Low to Medium V1

Manageable for a proof-of-value version. Lift increases when moving from Google Sheets and AppleScript into governed Microsoft 365 or enterprise integrations.

Future state

The future state is a practical operating layer that starts with a lightweight IT dashboard and matures into a governed department hub model connecting goals, work, meetings, decisions, evidence, KPIs, budget, and performance data across departments.

Department command centers

Each department eventually has a hub that shows current goals, initiatives, active work, key milestones, decisions needed, risks, budget status, KPIs, and linked evidence. V1 starts with the IT Team Hub and uses the current dashboard as the proving ground.

  • Department-specific dashboard view.
  • Standard intake and update workflows that write to a shared data source.
  • Meeting agenda and decision tracking.
  • Evidence log tied to goals, projects, and performance.
  • Automated rollups to leadership.

Organization-level rollup

Department hubs roll up to a COO, President, or executive view showing progress against annual priorities, cross-functional risks, dependencies, resourcing constraints, and decision bottlenecks.

  • Org-wide goal progress and initiative health.
  • Cross-department dependencies and blockers.
  • Decision center for leadership action.
  • Budget versus actuals and resource capacity views.
  • Performance evidence captured continuously instead of rebuilt at review time.

Path to implementation

The most practical V1 path is not a custom application or full Power Platform build. The recommended first release is a lightweight dashboard hosted from Commit GitHub, embedded into the SharePoint IT Team Hub, and updated through Google Sheets and AppleScript.

Phase Purpose Primary Work Output
1. Confirm V1 scope Define the first release as an IT dashboard embedded in the SharePoint IT Team Hub. Confirm dashboard sections, data fields, update cadence, owners, and what must remain manual versus automated. V1 scope, data dictionary, update workflow, and governance notes.
2. Create Commit GitHub host Move the dashboard from personal demo hosting into a Commit-owned GitHub repository. Create repo, upload HTML assets, establish publishing process, document file naming and update standards. Commit GitHub-hosted dashboard with controlled source files.
3. Build Google Sheets data source Use Google Sheets as the V1 backend for dashboard content. Create tabs for initiatives, milestones, KPIs, risks, decisions, budget, evidence, questions, kudos, intake, and updates. Structured Google Sheets workbook serving as the dashboard data source.
4. Connect intake and update forms Capture new work and progress updates directly into Google Sheets. Wire intake form and update form submissions to the correct Google Sheets tabs. Validate required fields and submission formatting. Forms that write directly to the dashboard source data.
5. Automate dashboard refresh Use AppleScript as the V1 automation layer to refresh the static dashboard HTML from Google Sheets. Script data export or transformation, update the dashboard file, validate output, and prepare the refreshed file for GitHub commit. Repeatable AppleScript refresh process.
6. Embed in SharePoint IT Team Hub Make the dashboard visible from the internal IT hub. Embed the GitHub-hosted dashboard link or iframe into SharePoint, test display, access, responsiveness, and user experience. IT Team Hub dashboard embedded in SharePoint.
7. Test, govern, and iterate Validate the workflow before broader rollout. Test data accuracy, update cadence, form submissions, script reliability, SharePoint display, and leadership usefulness. Stable V1 operating dashboard and backlog for V2 improvements.

What must be architected, built, prototyped, tested, and deployed

This work is feasible as a lightweight V1 if the build stays intentionally simple. The first version should prove value through GitHub hosting, Google Sheets data management, AppleScript refresh automation, and SharePoint embedding before moving to a more mature enterprise architecture.

Architect

  • Google Sheets data model for goals, initiatives, milestones, KPIs, risks, decisions, evidence, budget, questions, kudos, intake, and updates.
  • Relationship between form submissions, dashboard fields, and refreshed HTML output.
  • Commit GitHub repository structure and publishing workflow.
  • SharePoint IT Team Hub embed approach and access expectations.
  • Governance standards for who updates source data, runs scripts, reviews output, and commits dashboard changes.

Build

  • Commit GitHub repository with dashboard HTML files.
  • SharePoint IT Team Hub page section for the embedded dashboard.
  • Google Sheets workbook with structured tabs that match dashboard content.
  • Intake and update forms that write directly to Google Sheets.
  • AppleScript process that transforms Google Sheets data into refreshed dashboard content.

Prototype

  • IT Operations dashboard embedded in the SharePoint IT Team Hub.
  • Google Sheets source data populated with realistic IT portfolio records.
  • Working intake and update submissions into the Google Sheet.
  • AppleScript refresh run against sample data and then live pilot data.
  • Leadership review of the embedded dashboard experience.

Test

  • Form submission accuracy into Google Sheets.
  • Google Sheets field mapping into the dashboard HTML.
  • AppleScript reliability, error handling, and refresh timing.
  • GitHub Pages publishing and cache behavior after updates.
  • SharePoint embed display, responsiveness, and user experience.

Deploy

  • Commit GitHub-hosted dashboard page.
  • Embedded dashboard in the SharePoint IT Team Hub.
  • Documented runbook for updating the Google Sheet and running the AppleScript refresh.
  • Owner review process before dashboard changes are committed.
  • Backlog for V2 migration to SharePoint Lists, Power Automate, Power BI, or Power Apps if needed.

Operate

  • Weekly or biweekly Google Sheets data review.
  • Scheduled AppleScript refresh and GitHub publish cadence.
  • Change log for dashboard updates and data structure changes.
  • Quarterly goal and KPI refresh.
  • V2 decision checkpoint after the IT pilot proves value.

Dependencies, risks, and assumptions

The primary V1 challenge is keeping the workflow simple while still making it reliable. The dashboard can be practical quickly, but only if the Google Sheets structure, AppleScript process, GitHub publishing workflow, and SharePoint embed are governed clearly.

Dependencies

  • Commit-owned GitHub repository or approval to host the dashboard from a Commit-controlled GitHub location.
  • SharePoint IT Team Hub page where the dashboard can be embedded.
  • Google Sheets workbook used as the V1 source of truth.
  • Working intake and update forms that save to Google Sheets.
  • Mac environment capable of running the AppleScript refresh process.
  • Owner approval process before refreshed dashboard files are committed to GitHub.

Risks

  • AppleScript is useful for V1 but may create a single-user dependency if only one person can run or maintain it.
  • Google Sheets may create governance or access concerns if sensitive performance, budget, or evidence data is stored there.
  • Dashboard may become stale if the script is not run on a reliable cadence.
  • GitHub Pages caching or publishing delays may confuse users after updates.
  • SharePoint embed behavior may vary depending on tenant settings, browser behavior, and iframe restrictions.
  • Manual review is still required to prevent inaccurate or incomplete data from being published.

Assumptions

  • V1 can use Google Sheets as the data source even though SharePoint or Dataverse may be better for a later governed enterprise version.
  • Initial pilot uses IT Operations as the model department.
  • Commit GitHub hosting is approved or can be approved.
  • SharePoint can embed or link to the GitHub-hosted dashboard in a user-friendly way.
  • Manual review and human validation remain part of the workflow before publishing updates.
  • The first version should prioritize visibility, speed, and habit formation over full enterprise automation.

Feasibility and practical lift

The project is practical as a lightweight V1 because it uses tools that are already accessible: GitHub, SharePoint, Google Sheets, forms, and AppleScript. The lift is much lower than building a production app, but reliability and governance still need to be designed intentionally.

Recommendation: Build the first working version as a lightweight V1: Commit GitHub hosting, SharePoint IT Team Hub embed, Google Sheets as the data source, and AppleScript for dashboard refresh. This is practical, fast, and low-cost. Treat it as a proof-of-value pilot, then decide whether V2 should move into SharePoint Lists, Power Automate, Power BI, Power Apps, Dataverse, or another enterprise architecture.

Projected implementation timeline

This projection assumes a fast V1 rollout using Commit GitHub, SharePoint embedding, Google Sheets, intake/update forms, and AppleScript automation. Later phases move toward governed production architecture only after the V1 proves value.
